Ragrails exposes tool calling through one provider-neutral interface, so the same tool loop works across providers. This is the mechanism behind the interactive CLI's [agentic tools](/advanced/agentic-tools); this page covers the LLM layer itself.

<Note>**Agentic tools vs tool calling.** [Agentic tools](/advanced/agentic-tools) is the ready-made CLI experience (`web_fetch`, `api_call`, confirmation prompts). Tool calling is the underlying provider API you'd use to build your own tool loop.</Note>

## `complete_with_tools()`

Every LLM provider object has `complete_with_tools()`. You pass the conversation as a list of provider-neutral messages plus your tool schemas; it returns either text or a list of tool calls to run.

```python SDK
response = llm.complete_with_tools(
    messages=messages,        # provider-neutral message objects (below)
    system="You are a helpful assistant.",
    tools=[{                  # JSON-schema tool definitions
        "name": "get_weather",
        "description": "Get the weather for a city.",
        "input_schema": {
            "type": "object",
            "properties": {"city": {"type": "string"}},
            "required": ["city"],
        },
    }],
)

if response.tool_calls:
    for call in response.tool_calls:
        result = run_my_tool(call.name, call.arguments)
        # feed the result back as a ToolResultMessage, then call again
else:
    print(response.text)
```

## Message types

Provider-neutral dataclasses describe the tool loop. Ragrails translates them to each provider's wire format.

| Type | Fields | Role |
| --- | --- | --- |
| `ChatMessage` | `role`, `content` | A plain user or assistant message |
| `AssistantToolCallMessage` | `tool_calls`, `content` | The assistant's turn that requested tools |
| `ToolResultMessage` | `tool_call_id`, `content` | Your tool's output, fed back to the model |
| `ToolCall` | `id`, `name`, `arguments` | One requested call (`arguments` is parsed JSON) |
| `LLMToolResponse` | `text`, `tool_calls`, `input_tokens`, `output_tokens`, `model`, `provider` | The return value of `complete_with_tools()` |

A typical loop: send messages, get an `LLMToolResponse`, run any `tool_calls`, append an `AssistantToolCallMessage` and a `ToolResultMessage` per call, then call again until `tool_calls` is empty.

## Provider support

| Provider | Tool calling |
| --- | --- |
| OpenAI | Supported |
| Anthropic | Supported |
| Google (Gemini) | Not yet, `complete_with_tools()` raises `NotImplementedError` |

All Gemini models report `supports_tools = false` in the [model catalog](/reference/models), so pick an OpenAI or Anthropic model for any tool-using workflow.

<Tip>Use `first_tool_model()` or check a model's `supports_tools` before starting a tool loop, so you fail fast on a non-capable model.</Tip>
